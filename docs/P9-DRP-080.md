# P9-DRP-080: Disaster Recovery Plan

**Quantum Nexus Forge / Sentinel Forge — Cognitive AI Orchestration Platform**
**Neural Lattice Cognitive Architecture | Phase 9: Closure**

---

## 1. Purpose and Objectives

This Disaster Recovery Plan (DRP) establishes the procedures, responsibilities, and recovery targets for restoring the Quantum Nexus Forge / Sentinel Forge platform following any disruptive event that renders the system unavailable. The platform is a locally deployed cognitive AI orchestration tool built on Python FastAPI, SQLite, and React, running on a single personal machine.

**Objectives:**

- Define recovery procedures that restore full system functionality within 1 hour (RTO).
- Ensure no more than 24 hours of data loss through daily backups (RPO).
- Provide step-by-step runbooks with executable commands for each failure scenario.
- Establish a repeatable, testable recovery process that can be executed on any machine meeting minimum requirements.
- Document all assumptions, dependencies, and constraints so recovery can be performed even if the primary operator is unavailable.

## 5. Definitions

| Term | Definition |
|---|---|
| RTO (Recovery Time Objective) | The maximum acceptable duration between a disruption and the restoration of service. Target: **1 hour**. |
| RPO (Recovery Point Objective) | The maximum acceptable amount of data loss measured in time. Target: **24 hours**. |
| MTD (Maximum Tolerable Downtime) | The absolute maximum time the system can be unavailable before unacceptable impact occurs. Target: **48 hours**. |
| DR (Disaster Recovery) | The process of restoring IT systems, data, and operations after a disruptive event. |
| WAL (Write-Ahead Logging) | SQLite journaling mode that allows concurrent reads during writes and provides crash resilience. |
| Runbook | A documented set of step-by-step procedures for performing a specific recovery operation. |

## 9. Recovery Procedures / Runbooks

### Scenario 1: Machine Failure (Complete Loss of Primary Machine)

**Trigger:** Primary machine (ASUS Vivobook) is lost, stolen, destroyed, or unbootable.

**Prerequisites:** Recovery machine with Python 3.11+, Node.js (LTS), Git, and internet access.

**Procedure:**

```bash
# Step 1: Clone the repository from GitHub
git clone https://github.com/<owner>/Neural-Lattice-Architecture-and-Workflow-Optimization.git
cd Neural-Lattice-Architecture-and-Workflow-Optimization

# Step 2: Install Python dependencies
pip install -e ".[dev]"

# Step 3: Install frontend dependencies (if applicable)
cd frontend && npm install && cd ..

# Step 4: Restore SQLite database from backup
# Copy the most recent SQLite backup file to the expected database path
cp /path/to/backup/database.db ./data/database.db

# Step 5: Verify database integrity
python -c "import sqlite3; conn = sqlite3.connect('./data/database.db'); print(conn.execute('PRAGMA integrity_check').fetchone())"

# Step 6: Start the application
uvicorn app.main:app --host 0.0.0.0 --port 8000

# Step 7: Verify the frontend (in a separate terminal)
cd frontend && npm start
```

**Estimated recovery time:** 45–60 minutes

**Validation:** Access `http://localhost:8000/docs` (API) and `http://localhost:3000` (frontend). Confirm data is present and API endpoints respond.

---

### Scenario 2: Database Corruption

**Trigger:** SQLite database file is corrupted, unreadable, or returns errors on query.

**Procedure:**

```bash
# Step 1: Stop the running application
# Kill the uvicorn process (Ctrl+C or kill <pid>)

# Step 2: Verify corruption
python -c "import sqlite3; conn = sqlite3.connect('./data/database.db'); print(conn.execute('PRAGMA integrity_check').fetchone())"
# If output is anything other than ('ok',), the database is corrupted

# Step 3: Move the corrupted file aside for analysis
mv ./data/database.db ./data/database.db.corrupted.$(date +%Y%m%d%H%M%S)

# Step 4a: Restore from backup copy
cp /path/to/backup/database.db ./data/database.db

# Step 4b: (Alternative) If no backup is available, let the application create a fresh database
# The FastAPI application will auto-create an empty database on startup

# Step 5: Verify restored database integrity
python -c "import sqlite3; conn = sqlite3.connect('./data/database.db'); print(conn.execute('PRAGMA integrity_check').fetchone())"

# Step 6: Restart the application
uvicorn app.main:app --host 0.0.0.0 --port 8000
```

**Estimated recovery time:** 15–30 minutes

**Validation:** Run `PRAGMA integrity_check` returns `('ok',)`. Application starts without errors. Data queries return expected results.

---

### Scenario 3: Code Corruption or Bad Deployment

**Trigger:** Source code is accidentally modified, deleted, or a bad commit breaks functionality.

**Procedure:**

```bash
# Step 1: Identify the last known good commit
git log --oneline -20

# Step 2: Reset to the last known good commit
git reset --hard <commit-hash>

# Step 3: Reinstall dependencies (in case pyproject.toml changed)
pip install -e ".[dev]"

# Step 4: Restart the application
uvicorn app.main:app --host 0.0.0.0 --port 8000
```

**Estimated recovery time:** 10–15 minutes

**Validation:** Application starts without errors. API endpoints respond correctly. Frontend loads and displays data.

---

### Scenario 4: Dependency Failure (PyPI/npm Package Unavailable or Broken)

**Trigger:** A Python or Node.js dependency fails to install or introduces a breaking change.

**Procedure:**

```bash
# Step 1: Check pyproject.toml for pinned versions
cat pyproject.toml | grep -A 50 "dependencies"

# Step 2: If a dependency broke after update, revert to pinned version
# Edit pyproject.toml to pin the problematic package to the last known working version
# Example: change "somepackage>=1.0" to "somepackage==1.0.3"

# Step 3: Clear pip cache and reinstall
pip cache purge
pip install -e ".[dev]"

# Step 4: For frontend dependency issues
cd frontend
rm -rf node_modules package-lock.json
npm install
cd ..

# Step 5: Restart and verify
uvicorn app.main:app --host 0.0.0.0 --port 8000
```

**Estimated recovery time:** 15–30 minutes

**Validation:** `pip install` completes without errors. `npm install` completes without errors. Application starts and all features function.

---

### Scenario 5: GitHub Repository Loss

**Trigger:** Remote GitHub repository is deleted, corrupted, or inaccessible.

**Procedure:**

```bash
# Step 1: Verify the local clone is intact
cd Neural-Lattice-Architecture-and-Workflow-Optimization
git status
git log --oneline -5

# Step 2: Create a new GitHub repository (via CLI or web UI)
gh repo create Neural-Lattice-Architecture-and-Workflow-Optimization --private --source=. --remote=origin

# Step 3: Push all branches and tags to the new remote
git push --all origin
git push --tags origin

# Step 4: Verify the remote repository
gh repo view
```

**Estimated recovery time:** 15–30 minutes

**Validation:** `git remote -v` shows the new remote. `git push` succeeds. Repository is visible on GitHub.

### 9.1 DR Invocation Criteria

| Criterion | Threshold | Action |
|---|---|---|
| Primary machine unresponsive | Cannot boot or access within 30 minutes | Invoke Scenario 1 (Machine Failure) |
| Database read/write errors | Any `PRAGMA integrity_check` failure | Invoke Scenario 2 (Database Corruption) |
| Application fails to start | Startup failure persists after 2 restart attempts | Invoke Scenario 3 (Code Corruption) or Scenario 4 (Dependency Failure) |
| Dependency installation failure | `pip install` or `npm install` fails on clean environment | Invoke Scenario 4 (Dependency Failure) |
| GitHub repository inaccessible | GitHub returns 404 or access denied for >1 hour | Invoke Scenario 5 (GitHub Repository Loss) |
| Data loss detected | Missing or inconsistent records in SQLite | Invoke Scenario 2 (Database Corruption) |

## 12. Backup, Replication, and Data Protection

| Data Asset | Backup Method | Backup Location | Frequency | Retention | Encryption | Responsible |
|---|---|---|---|---|---|---|
| Application source code | `git push` to GitHub | GitHub (remote) | Daily (at minimum, on every meaningful change) | Indefinite (full git history) | HTTPS/SSH in transit; GitHub at-rest encryption | Shannon Brian Kelly |
| SQLite database file | Manual file copy (`cp database.db <backup-path>`) | External USB drive or secondary local directory | Daily | Last 7 copies (rolling) | None (apply GPG encryption if storing off-machine) | Shannon Brian Kelly |
| SQLite WAL file | Copied alongside database file during checkpoint | Same as database backup | Daily | Same as database | Same as database | Shannon Brian Kelly |
| Configuration / .env files | Stored in password manager or encrypted archive | Separate from code repository | On every change | Current version | Password manager encryption | Shannon Brian Kelly |
| Documentation (docs/) | `git push` to GitHub (tracked in repository) | GitHub (remote) | Daily | Indefinite (full git history) | HTTPS/SSH in transit | Shannon Brian Kelly |

**Replication:** Not applicable. The system uses a single SQLite file on a single machine. No replication architecture exists.

**Data protection notes:**
- SQLite WAL mode provides crash resilience but not backup. Always run `PRAGMA wal_checkpoint(TRUNCATE)` before copying the database file to ensure a consistent backup.
- Never commit the `.env` file or database file to the git repository.
